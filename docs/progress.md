## Pivots / direction changes

## Host-language reviews

- _(2026-05-20)_ Re-confirmed mid-Phase 1 (after the Bicep docs walker slice 1 shipped). Conclusion: **stay on PowerShell, no Option 3 trigger has fired.** Friction observed this turn — strict-mode 3.0 quirks (`.PSObject.Properties.Name` on empty pscustomobjects, `$arr[-1]` ambiguity, inline-if `@()` vs `@($x)` type-mixing), Pester `<word>` placeholder trap in `It` titles, backtick literalness in single-quoted regex — was all *language quirks*, not *language fit*. Solved once per pattern and codified (use `[List[string]]::new()` + explicit `[$list.Count - 1]` indexing; use `.PSObject.Properties['name']` indexer instead of `.Name` projection; rename `It` titles that contain `<word>` substrings; count backticks in single-quoted regex literally). The Bicep walker we just shipped is three in-process function calls (`Convert-AvmBicepToArm` → `Format-AvmBicepOutputsSection` → `Merge-AvmReadmeSection`); in C# every one of those would be a `pwsh.exe` shell-out or a `Microsoft.PowerShell.SDK` runspace. None of the documented Option-3 triggers (argv parser ≥2×, polished TUI, startup latency dominates, Windows SIGINT blocker, concurrency runspaces can't service) have fired. Next reconsideration point flagged: the recursive UDT walker in the upcoming Parameters-section slice — if discriminated-union or `$ref` cycle handling gets error-prone, that's where weak typing will start to hurt.

**3 of 4 don't fit the existing lock model.** The consolidation plan flags this — `docs/avm-consolidation-plan.md` §11 open question 2 ("Fork strategy for `lonegunmanb/avmfix` and `matt-FFFFFF/porch`") — but no resolution is recorded. Three credible options:

**Option A — Build-and-host pattern.** A new CI workflow in this repo (or a sibling release-only repo) cross-compiles `avmfix` / `mapotf` / `grept` from their upstream tags for 6 platforms, signs + checksums, and publishes as GitHub Releases under `Azure/*`. `tools.lock.psd1` entries pin to those URLs + SHA256s. The existing `Resolve-AvmTool` / `Install-AvmTool` code path stays unchanged.
- **Pros**: install model unchanged; trustworthy provenance; deterministic install (offline-cacheable, SHA-verifiable, lock-pinned).
- **Cons**: ongoing release pipeline maintenance; must track each upstream tag; need agreement with upstream maintainers (or accept "soft fork" with our own release cadence); new GitHub Actions surface.
- **Effort**: one new workflow, one new lock entry per tool, ~2 sub-slices per tool (CI + lock).

**Option B — Native `go install` path in `Install-AvmTool`.** Add a `goModule` lock entry shape (`module`, `version`, `entrypoint`) that runs `go install <module>@<version>` into the managed cache, then resolves the resulting binary. `Resolve-AvmTool` learns a second install kind.
- **Pros**: zero coordination with upstream; minimal lock-schema changes; per-tool version pinning still works via the Go module proxy.
- **Cons**: introduces a **Go-toolchain prerequisite** (`go` must be on `$PATH` at install time, both locally and in CI); deterministic install model weakens (no SHA verification of the built binary against a known-good hash — supply-chain integrity relies on Go module proxy + `sum.golang.org` checksum DB); spec §10 currently assumes URL-downloadable archives, so it would need an explicit amendment.
- **Effort**: one new lock-entry kind, one new install branch in `Install-AvmTool`, doctor-step extension to check for `go` on `$PATH`, spec §10 amendment.

**Option C — Defer Phase 2 §2 engine implementations entirely.** Pivot to Phase 2 §3 (validation outward — integration tests against real modules using whatever the user has installed). Document that `transform` / `check convention` / `format`-`avmfix`-chain still require the legacy `./avm` wrapper (or `mcr.microsoft.com/azterraform`) until the architectural call is made.
- **Pros**: avoids the decision; unblocks `avm check policy` (conftest is ready) and demoability via the wired `pre-commit`/`pr-check` chains for `format` / `lint` / `test` / `docs`.
- **Cons**: doesn't deliver on Phase 2's "no container required" promise for those three steps; pushes the call to Phase 3+.

**A/B hybrid is possible**: ship Option B so the CLI is self-contained on dev workstations with Go available, AND adopt Option A long-term for CI environments that don't pre-install Go. The lock-entry kind would carry both `goModule` (preferred install) and `urlTemplate` (fallback / hash-pinned alternate).

**No decision recorded yet.** The next implementation slice for `avm transform`, `avm check convention`, or the `avm format` `avmfix`-chain is blocked on this call. `avm check policy` and the pinned-asset feature can proceed independently.
